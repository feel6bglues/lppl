## Phase 1: P0 紧急修复

### FIX-003: `_risk_band` 风险分层放宽

**对应 ISSUE**: ISSUE-003

**修复目标**: 使 risk_band 能够区分"负超额但回撤可控"与"负超额且回撤失控"的候选

**修复思路**:

当前逻辑：`annualized_excess_return <= 0` 直接 DANGER。这导致高弹性组所有候选（均负超额）无法区分。

新逻辑引入分层：
1. **极端 DANGER**：回撤 > 40% 或（超额 < -5% 且交易 < 3 次）
2. **普通 Warning**：超额为负但不极端
3. **Watch/Safe**：超额为正

**具体修改**:

```python
# 文件: src/investment/tuning.py
# 位置: _risk_band() 函数, 约第 42-53 行

# 修改前:
def _risk_band(row: pd.Series) -> str:
    max_drawdown = float(row.get("max_drawdown", 0.0))
    annualized_excess_return = float(row.get("annualized_excess_return", 0.0))
    calmar_ratio = float(row.get("calmar_ratio", 0.0))

    if max_drawdown <= -0.25 or annualized_excess_return <= 0.0:
        return "DANGER"
    if max_drawdown <= -0.18 or calmar_ratio < 0.20:
        return "Warning"
    if max_drawdown <= -0.10 or calmar_ratio < 0.50:
        return "Watch"
    return "Safe"

# 修改后:
def _risk_band(row: pd.Series) -> str:
    max_drawdown = float(row.get("max_drawdown", 0.0))
    annualized_excess_return = float(row.get("annualized_excess_return", 0.0))
    calmar_ratio = float(row.get("calmar_ratio", 0.0))
    trade_count = float(row.get("trade_count", 0.0))

    # 第一层：极端风险（不可接受）
    if max_drawdown <= -0.40:
        return "DANGER"
    # 第二层：高回撤 或 "无为策略"（负超额 + 低交易频率）
    if max_drawdown <= -0.25 or (annualized_excess_return <= -0.05 and trade_count < 3):
        return "DANGER"
    # 第三层：负超额但不极端 → 降级为 Warning
    if annualized_excess_return <= 0.0:
        return "Warning"
    # 第四层：正超额但回撤/Calmar 不佳
    if max_drawdown <= -0.18 or calmar_ratio < 0.20:
        return "Warning"
    if max_drawdown <= -0.10 or calmar_ratio < 0.50:
        return "Watch"
    return "Safe"
```

**分层对照表**:

| 场景 | 修改前 | 修改后 | 理由 |
|------|--------|--------|------|
| 超额 -1.7%, 回撤 -28%, 交易 6 次 | DANGER | Warning | 负超额但回撤可控、交易活跃 |
| 超额 -5.2%, 回撤 -45%, 交易 1 次 | DANGER | DANGER | 双重恶化 |
| 超额 -0.5%, 回撤 -20%, 交易 8 次 | DANGER | Warning | 轻微负超额，其他指标良好 |
| 超额 +1.8%, 回撤 -24%, 交易 9 次 | Warning | Warning | 正超额但回撤偏大 |
| 超额 +2.6%, 回撤 -15%, 交易 10 次 | Watch | Safe | 各项指标良好 |

**影响范围**:
- 仅影响 `risk_band` 和 `suggest_position` 列
- 不影响 `eligible` 判断和 `objective_score` 计算
- 高弹性组将从"全部 DANGER"变为"大部分 Warning"

**风险评估**: 低。仅影响风险标签，不改变核心排序。

### FIX-008: `turnover_cap` 动态调整

**对应 ISSUE**: ISSUE-010

**修复目标**: 快 MA 组合使用更宽松的 turnover 门槛

**修复思路**:

不直接修改 `turnover_cap`，而是在评分中给予 turnover 适当权重（当前已有 10%），同时提高门槛至 10.0。

**具体修改**:

```python
# 文件: src/investment/group_rescan.py
# 位置: BALANCED_PLAN, 约第 59 行

# 修改前:
turnover_cap=8.0,

# 修改后:
turnover_cap=10.0,
```

**理由**: 快 MA 组合（MA10/120）天然交易频率更高，8.0 门槛基于慢 MA（MA20/250）经验值。提高至 10.0 允许快 MA 候选通过硬门槛，同时评分中 turnover 权重（10%）仍会惩罚过度交易。

### FIX-004: 分组聚合增加 worst-case 指标

**对应 ISSUE**: ISSUE-008

**修复目标**: 评分系统能看到组内最差表现，而非仅看均值

**修复思路**:

当前聚合只用 `mean()`，导致"一只 +3%、一只 -1%"平均为 +1% 的假象。

新增 worst-case 指标：
- `min_excess`：组内最小超额回报
- `max_drawdown_worst`：组内最大回撤（最差的）
- `min_trade_count`：组内最少交易次数

**具体修改**:

```python
# 文件: src/investment/group_rescan.py
# 位置: summarize_rescan_results() 函数, 约第 282-308 行

# 修改前:
summary_df = (
    raw_df.groupby([...], as_index=False)
    .agg(
        eligible_count=("eligible", "sum"),
        symbol_count=("symbol", "nunique"),
        avg_excess=("annualized_excess_return", "mean"),
        avg_drawdown=("max_drawdown", "mean"),
        avg_trade_count=("trade_count", "mean"),
        avg_turnover=("turnover_rate", "mean"),
        avg_whipsaw=("whipsaw_rate", "mean"),
        avg_calmar_ratio=("calmar_ratio", "mean"),
    )
)

# 修改后:
summary_df = (
    raw_df.groupby([...], as_index=False)
    .agg(
        eligible_count=("eligible", "sum"),
        symbol_count=("symbol", "nunique"),
        # 均值指标（保留）
        avg_excess=("annualized_excess_return", "mean"),
        avg_drawdown=("max_drawdown", "mean"),
        avg_trade_count=("trade_count", "mean"),
        avg_turnover=("turnover_rate", "mean"),
        avg_whipsaw=("whipsaw_rate", "mean"),
        avg_calmar_ratio=("calmar_ratio", "mean"),
        # 新增 worst-case 指标
        min_excess=("annualized_excess_return", "min"),
        max_drawdown_worst=("max_drawdown", "min"),
        min_trade_count=("trade_count", "min"),
    )
)
```

**评分系统适配**:

```python
# 文件: src/investment/group_rescan.py
# 位置: summarize_rescan_results() 函数, 约第 312-329 行

# 修改 score_signal_tuning_results 调用，增加 worst-case 列
scored = score_signal_tuning_results(
    summary_df.rename(
        columns={
            "avg_excess": "annualized_excess_return",
            "avg_drawdown": "max_drawdown",
            # ... 其他重命名
        }
    ),
    ...
)

# 在 scored 结果中增加 worst-case 惩罚
# 如果组内最小超额为负，扣减 objective_score
if "min_excess" in scored.columns:
    negative_min_penalty = scored["min_excess"].clip(upper=0).abs() * 0.1
    scored["objective_score"] = scored["objective_score"] - negative_min_penalty
```

**影响范围**:
- 新增 3 列到 summary.csv
- `objective_score` 计算增加 worst-case 惩罚
- 平衡组中"一只达标一只不达标"的候选排名会下降

**风险评估**: 中。改变评分逻辑，需要重新运行分组扫描验证。

### FIX-005: `hard_reject` 逻辑强化

**对应 ISSUE**: ISSUE-009

**修复目标**: 确保 `eligible_count=0` 的候选不会排在 `eligible_count>0` 之前

**修复思路**:

当前 `hard_reject=False` 导致不合格候选仍参与评分排名。改为：
1. 在 `summarize_rescan_results` 中对 `eligible_count=0` 的候选设置 `objective_score = -1`
2. 排序时优先 `eligible_count`，再 `objective_score`

**具体修改**:

```python
# 文件: src/investment/group_rescan.py
# 位置: summarize_rescan_results() 函数末尾, 约第 330-334 行

# 修改前:
scored["group_pass"] = scored["eligible_count"] >= plan.pass_threshold
return scored.sort_values(
    ["group_pass", "eligible_count", "objective_score", "annualized_excess_return", "max_drawdown"],
    ascending=[False, False, False, False, False],
).reset_index(drop=True)

# 修改后:
scored["group_pass"] = scored["eligible_count"] >= plan.pass_threshold

# 对 eligible_count=0 的候选，强制设置最低分数
scored.loc[scored["eligible_count"] == 0, "objective_score"] = scored.loc[
    scored["eligible_count"] == 0, "objective_score"
].clip(upper=-0.01)  # 确保低于任何合格候选

return scored.sort_values(
    ["group_pass", "eligible_count", "objective_score", "annualized_excess_return", "max_drawdown"],
    ascending=[False, False, False, False, False],
).reset_index(drop=True)
```

**影响范围**:
- 高弹性组所有候选（`eligible_count=0`）排名会统一靠后
- 排序逻辑更清晰：先看合格数，再看质量

**风险评估**: 低。仅影响排序，不改变候选本身的质量评估。

## 回滚策略

### 回滚原则

1. **每个 Phase 独立 commit**，便于单独回滚
2. **保留修改前的输出文件**，用于数值对比
3. **回测结果可复现**：所有随机种子固定

### 回滚步骤
